# 에이전트 워크플로 운영 가이드

## 요약(핵심 결론)

- 이 문서는 역할 기반 운영의 단일 기준 문서다.
- 역할은 `PM(최종 의사결정)`, `TL(기술 승인/리스크 게이트)`, `Subagent(실행)`으로 분리한다.
- 기본 흐름은 `Issue -> Plan -> 실행 -> PM 리뷰 -> TL 리뷰 -> PM 최종 승인 -> trunk`다.
- 연관 이슈가 없으면 태스크를 시작하지 않고 `blocked` 상태로 유지한다.

## 실행/적용 방법

1. 이슈 게이트 확인
- 모든 태스크는 연관 이슈 번호가 있어야 시작할 수 있다.
- 연관 이슈가 없으면 상태를 `blocked`로 두고, 이슈 생성 후 `todo`로 전환한다.

2. PM 계획 수립
- PM은 목표, 범위(in/out), DoD, 의존성을 명시한다.
- PM은 context packet을 배포한다.
- context packet 구성:
  - 핵심 문서 최대 3개
  - 현재 결정사항 요약
  - 금지사항/주의사항

3. 실행과 상태 전이
- 표준 상태: `blocked -> todo -> in_progress -> review_pm -> review_tl -> done`
- Subagent 결과물은 반드시 아래를 포함한다.
  - 변경 요약
  - 검증 결과(테스트/체크)
  - 리스크
  - 미해결 항목

4. 리뷰와 승인
- PM 리뷰: 요구사항 적합성, 범위 준수, 완료조건 충족 여부 확인
- TL 리뷰: 정확성, 회귀/안전성, 테스트 충분성 확인
- PM이 TL 결과를 반영해 최종 승인하고 trunk 진행 여부를 결정한다.

## 세부 스펙/예외

### Subagent 사용 기준

- 독립적으로 분리 가능한 작업 단위인가
- 병렬 처리로 리드타임 단축 효과가 있는가
- 결과 검증 기준(DoD/테스트)이 명확한가
- 리스크가 커서 역할 분리 리뷰가 필요한가

### Subagent 비권장 기준

- 한 파일/한 함수 수준의 소규모 수정
- 여러 에이전트가 동시에 같은 파일을 수정해야 하는 강결합 작업
- 조정 비용이 실제 작업 비용보다 큰 작업

### PM/TL 충돌 규칙

- 기본 원칙: PM 판단 우선
- 예외: TL이 아래 범주의 "명백한 위험"을 증거와 함께 제시하면 TL override를 따른다.
- 명백한 위험 범주:
  - 보안 취약점
  - 데이터 손상/유실 가능성
  - 정책/법 위반 가능성
  - 치명 회귀(핵심 기능 실패)
  - 테스트 부재로 품질 미보장
