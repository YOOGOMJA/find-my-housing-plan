# LH/SH 도메인 지식 학습 로드맵

작성일: 2026-03-07

## 1) 목적

이 문서는 SH/LH 공고 자동 판정 기능을 구현하기 전에, 반드시 이해해야 하는 도메인 지식을 단계적으로 학습하기 위한 로드맵이다.

핵심 원칙:

- 구현 전에 용어/기준/판정 논리를 먼저 고정한다.
- "법령/고시 원문 > 공고문 원문 > 포털 요약" 우선순위를 지킨다.
- 각 항목은 `완료 기준`을 만족해야 다음 단계로 넘어간다.

## 1-1) 이 문서를 언제 쓰는가

- 처음 도메인에 진입할 때: 무엇을 먼저 이해해야 하는지 순서를 잡을 때
- 구현 직전: 내가 필요한 도메인 준비를 끝냈는지 점검할 때
- 리뷰 시점: 특정 판정 로직이 왜 필요한지 역으로 추적할 때

## 1-2) 이 문서가 다루지 않는 것

- 실제 코드 파일 구조나 구현 진행 상태
- 특정 API 응답 스키마의 최신 변경사항
- 연도별 수치 테이블의 최신 값 자체

위 항목은 별도 설계 문서, 구현 코드, 외부 원문 검증으로 확인한다.

## 2) 학습 단계 요약

1. 기준 체계 이해: 소득/자산/무주택/청약통장의 정의와 판정 기준일
2. 사업 유형 이해: 임대유형별 목적·대상·예외
3. 공고 해석 능력: 공고문에서 실제 판정 근거 문장 추출
4. 데이터 모델 연결: 지식 항목을 스키마 필드로 변환

## 3) 상세 학습 항목

### 3-1. 소득 기준 체계

- 왜 필요한가: 소득기준 계산이 틀리면 전체 판정이 무효가 된다.
- 알아야 할 것:
  - 소득 기준의 모수(월평균, 세대/가구 범위, 세전/세후 해석)
  - 가구원수별 기준표 적용 방식
  - 공고일 기준 적용 여부
- 어디서 확인할까:
  - 마이홈 자격 안내: https://www.myhome.go.kr/hws/portal/cont/selectNHTQualificationView.do
  - 개별 공고문(공고일 기준 문구)
- 완료 기준:
  - "1인/2인/3인 가구" 기준 계산 예시를 직접 3개 만들어 설명할 수 있다.

### 3-2. 자산 기준 체계

- 왜 필요한가: 소득 통과자도 자산/자동차 기준에서 탈락할 수 있다.
- 알아야 할 것:
  - 총자산/자동차가액 포함 범위
  - 사업유형별 기준 차이
- 어디서 확인할까:
  - 마이홈 자격 안내: https://www.myhome.go.kr/hws/portal/cont/selectNHTQualificationView.do
  - 공고문 자격 표
- 완료 기준:
  - 자산/자동차 기준이 다른 공고 2건을 비교해 차이점을 설명할 수 있다.

### 3-3. 무주택 요건과 기간

- 왜 필요한가: 우선공급/순위 판정에서 핵심 분기값이다.
- 알아야 할 것:
  - 무주택세대구성원 개념
  - 기간 계산 기준일(보통 공고일)
- 어디서 확인할까:
  - 주택공급 관련 법령: https://www.law.go.kr/LSW/lsStmdInfoP.do?lsiSeq=46188
  - LH 공고문 예시: https://apply.lh.or.kr/lhapply/apply/wt/wrtanc/selectWrtancInfo.do?panId=0000060987&ccrCnntSysDsCd=02&uppAisTpCd=05&aisTpCd=05&mi=1027
- 완료 기준:
  - 기준일이 바뀌었을 때 무주택기간 계산 결과가 어떻게 달라지는지 사례 2개를 설명할 수 있다.

### 3-4. 청약통장(입주자저축) 요건

- 왜 필요한가: 순위/당첨 가능성에 직접 영향이 있다.
- 알아야 할 것:
  - 통장 유형, 가입기간, 납입횟수, 순위 판정
  - 지역/유형별 요구 조건 차이
- 어디서 확인할까:
  - 청약Home: https://applyhome.co.kr/
  - 마이홈 자격 안내: https://www.myhome.go.kr/hws/portal/cont/selectNHTQualificationView.do
- 완료 기준:
  - 입력 필드(`가입일/납입횟수/통장유형`)만으로 1순위/2순위 예시 판정을 3개 수행할 수 있다.

### 3-5. 사업유형별 기본 룰

- 왜 필요한가: 같은 "임대"라도 유형마다 대상/자격이 다르다.
- 알아야 할 것:
  - 영구/국민/행복/장기전세/매입/전세/재개발 임대 차이
  - 유형별 우선공급/특별공급 분기
- 어디서 확인할까:
  - 마이홈 임대유형: https://m.myhome.go.kr/hws/portal/cont/selectContRentalHouseView.do
  - 서울주거포털 SH 유형: https://housing.seoul.go.kr/
- 완료 기준:
  - 유형 7개를 `목적/핵심자격/수집필드` 3열 표로 정리할 수 있다.

### 3-6. 공고 이벤트 구조(사전청약/본청약)

- 왜 필요한가: 동일 사업을 서로 다른 공고 단계로 관리해야 중복/누락을 막는다.
- 알아야 할 것:
  - 사전청약과 본청약의 단계 관계
  - 공고 ID 매핑 규칙
- 어디서 확인할까:
  - LH 사전청약 API: https://www.data.go.kr/data/15124599/openapi.do
  - LH 사전청약 공급 API: https://www.data.go.kr/data/15124601/openapi.do
  - LH 사전청약 상세 API: https://www.data.go.kr/data/15124603/openapi.do
- 완료 기준:
  - 동일 단지의 사전청약/본청약 공고를 연결하는 키 전략을 문서로 설명할 수 있다.

### 3-7. 기관별 데이터 특성(LH vs SH)

- 왜 필요한가: 수집 파이프라인 설계가 기관별로 달라진다.
- 알아야 할 것:
  - LH: API 중심 수집(목록/상세 분리)
  - SH: 공고문/사이트 중심 + 파일데이터 보조
- 어디서 확인할까:
  - LH API: https://www.data.go.kr/data/15058530/openapi.do
  - SH 데이터: https://www.data.go.kr/data/15122124/fileData.do
  - SH 사이트: https://www.i-sh.co.kr/index.do
- 완료 기준:
  - LH/SH 각각에 대해 "수집 1차/2차/3차" 절차를 5줄 이내로 설명할 수 있다.

### 3-8. 사용자 프로필 정보(결혼/나이/거주/세대/신생아)

- 왜 필요한가: 소득·자산 조건이 맞아도, 세대/거주/혼인/자녀 조건에서 최종 자격이 달라진다.
- 알아야 할 것:
  - 공고일 기준으로 평가되는 항목(나이, 거주기간, 혼인기간, 자녀연령)
  - 사용자 입력 필드와 파생 필드(예: `거주기간 = 기준일 - 전입일`) 분리
  - API가 아닌 사용자 입력/증빙 기반으로 수집해야 하는 항목 구분
- 어디서 확인할까:
  - 마이홈 신혼부부 특별공급: https://m.myhome.go.kr/hws/portal/cont/selectNewHomeShareTypeView.do
  - LH 공고 예시(전입일/태아/직계비속 문구): https://apply.lh.or.kr/lhapply/apply/wt/wrtanc/selectWrtancInfo.do?panId=2015122300018145&ccrCnntSysDsCd=03&uppAisTpCd=13&aisTpCd=26&mi=1026
- 완료 기준:
  - 결혼 여부/생년월일/전입일/세대원/신생아 여부를 입력받아 공고일 기준 파생값을 계산하는 규칙을 문서로 설명할 수 있다.

## 4) 지식 -> 구현 연결 규칙

학습한 지식은 아래 출력물로 연결해야 한다.

주의: 아래 출력물은 이 문서 안에 모두 포함되어 있다는 뜻이 아니라, 구현 시작 전 팀이 별도로 준비하거나 확정해야 하는 산출물 목록이다.

1. 용어 사전 (`business_type`, `supply_track`, `applicant_group` 등)
2. 판정 엔진 입력 스키마 (`user_profile`, `notice_rule`, `housing_info`)
3. 판정 사유 코드 사전 (`income_exceed`, `asset_exceed`, `subscription_rank_fail` 등)
4. 공고 파서 추출 항목 목록(필수/선택/판정불가 트리거)

## 5) 완료 정의 (Definition of Ready)

아래 5개를 모두 만족하면 구현 시작 가능 상태로 본다.

- 소득/자산/무주택/통장 기준을 공고 3건에 적용해 수동 판정 가능
- 사업유형 7개를 혼동 없이 분류 가능
- 사전청약/본청약 이벤트를 데이터 모델로 분리 설명 가능
- LH/SH 수집 파이프라인 차이를 문서로 설명 가능
- 필수 정보 누락 시 `판정불가` 처리 기준 합의 완료

## 6) 사용 방법

1. 먼저 `2) 학습 단계 요약`만 읽고 전체 흐름을 이해한다.
2. 실제 작업 범위에 맞는 `3) 상세 학습 항목`만 선택해서 본다.
3. 작업 시작 전에는 `4) 지식 -> 구현 연결 규칙`과 `5) 완료 정의`만 다시 확인한다.
