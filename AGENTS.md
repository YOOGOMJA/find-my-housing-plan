## 에이전트 응답 규칙

- 모든 응답과 답장은 기본적으로 한글로 작성한다.
- 사용자가 질문 의도를 밝혔거나 확인 질문이 필요한 경우, 현재 모드에서 사용 가능한 질문 전용 도구를 우선 사용한다.
- 질문 전용 도구가 없거나 현재 모드에서 사용할 수 없으면 일반 텍스트 질문으로 대체한다.
- Codex 환경에서는 Plan 모드일 때 `request_user_input`을 우선 사용하고, Default 모드에서는 일반 텍스트 질문을 사용한다.
- Claude 환경에서는 `AskUserQuestion` 계열 도구가 노출된 경우 이를 우선 사용하고, 해당 도구가 없으면 일반 텍스트 질문으로 대체한다.

### 태스크 목록 관리 규칙

- 작업을 계획/진행할 때는 태스크 목록을 생성하고 단계별로 상태를 갱신한다.
- 특정 도구명(`TaskCreate`, `TaskUpdate`)을 공통 표준으로 강제하지 않는다.
- 도구명이 환경별로 다르더라도 원칙은 동일하게 유지한다.
- 태스크 도구가 없으면 일반 텍스트로 할 일/진행/완료 상태를 같은 수준으로 기록한다.
- 상세 역할/리뷰/충돌/게이트 규칙은 `docs/agent-workflow.md`를 단일 기준으로 따른다.
- 작업 시작 시 PM은 context packet(핵심 문서 최대 3개, 현재 결정사항 요약, 금지사항)를 먼저 배포한다.
- 연관 이슈가 없으면 태스크 상태를 `blocked`로 두고, 이슈 생성 후에만 `todo`로 전환한다.

## 문서 작성/리뷰 원칙

- 문서를 작성할 때는 **progressive disclosure(점진적 공개)** 관점으로 구성한다.
- 문서 리뷰를 할 때도 같은 원칙을 적용해, 상위 요약 → 세부 내용 → 참고 자료 순으로 정보가 자연스럽게 드러나는지 확인한다.
- 기본 구조 권장: `요약(핵심 결론)` → `실행/적용 방법` → `세부 스펙/예외` → `참고 링크`.

## 도메인 문서 우선순위

- SH/LH 주택 공고, 청약, 자격 판정, 사업/공고 구분, 수집 필드, 판정 기준일처럼 도메인 지식이 필요한 작업은 먼저 `docs/domain/`을 확인한다.
- `docs/domain/`은 도메인 기준과 운영 정의를 확인하는 1차 문서다.
- `docs/references/`는 외부 링크, API, 사이트, 데이터 소스 같은 참고 자료를 보관하는 보조 문서다.
- 도메인 해석이 필요한 경우 우선순위는 `docs/domain/` → `docs/references/` → 외부 원문 확인 순서로 적용한다.

## 협업 규칙 (Commit / Issue / PR)

- 상세 규칙은 `CONTRIBUTING.md`를 기준으로 따른다.
- 브랜치 네이밍은 `CONTRIBUTING.md`의 `issue-<번호>` 규칙을 따른다.
- 작업 시작 전 GitHub Issue를 생성하고, 필요하면 유사/중복 이슈여도 현재 작업 추적용으로 만든다.
- 커밋은 Conventional Commits 형식을 따르며 `scope`는 이슈 번호(`(#123)`)를 사용한다.
- 커밋 메시지, 이슈, PR 제목/본문은 기본적으로 한글로 작성한다.
